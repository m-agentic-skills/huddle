# The 80% Problem — And What LLMs Actually Changed

There's a pattern every experienced engineer has seen.

You start building:

- APIs come together
- Flows start working
- UI looks complete

Very quickly, it feels like:

> 80% of the system is done.

Everyone relaxes.

It looks like progress. It feels like momentum.

---

## This is where things go wrong

In software development, there's a commonly observed pattern:

> "The last 20% takes 80% of the time."
>
> Reference: https://vadimkravcenko.com/shorts/project-estimates/

---

## What that actually means

Software doesn't get built linearly.

It gets built in two very different phases.

**Phase 1 — Fast, visible progress**

- Features appear quickly
- Everything seems to work
- The system feels "almost done"

**Phase 2 — Slow, invisible complexity**

- Edge cases start appearing
- Assumptions begin to break
- Real-world behavior shows up

And this second phase?

That's where systems either become solid — or fall apart.

---

## Now enter LLMs

With tools like Claude, the first phase has been compressed dramatically.

You can:

- Generate large parts of a system instantly
- Move from idea to working code in minutes
- Get structured, confident answers on demand

---

## So what changed?

You now reach:

> "80% done" — faster than ever before.

---

## What didn't change

The hardest part.

Because the last 20% is not about writing code.

It's about:

- Uncovering hidden assumptions
- Identifying failure paths
- Understanding real usage
- Validating decisions under pressure

---

## And here's the real shift

Before LLMs, the bottleneck was:

> "Can we build this?"

Now the bottleneck is:

> "Did we think about this deeply enough?"

---

## The real problem isn't answers

Answers are easy now.

Anyone can generate:

- Code
- Architecture
- Documentation

But:

> The ability to ask the right questions is becoming rare.

---

## What questions actually matter

The uncomfortable ones:

- What happens when this fails in production?
- Who controls this input?
- What assumption are we making here?
- Is this solving the real problem, or just a version of it?

These are the questions that define whether a system survives.

---

## Real systems are not built in one voice

- A developer sees implementation
- A manager sees feasibility
- A security engineer sees risk
- A product person sees intent

Each one asks different questions.

Most of those questions come too late.

---

## What changes when questions come early

Everything.

- Missing conditions surface before production
- Tradeoffs become explicit
- Decisions become intentional
- Communication becomes clearer
